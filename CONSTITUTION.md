# CONSTITUTION.md - AI 行为准则

> 本文档规定 AI 在此项目中应遵循的行为准则和约束。
>
> 最后更新：2026-04

## 核心原则

### 1. 代码质量优先

- **必须** 生成类型安全的 TypeScript 代码
- **必须** 遵循项目现有的编码风格和模式
- **必须** 确保代码通过 ESLint 检查
- **禁止** 使用 `any` 类型，除非有明确理由
- **禁止** 生成未经测试的复杂逻辑

### 2. 遵循项目规范

- **必须** 使用 `<script setup lang="ts">` 语法
- **必须** 使用 Composition API
- **必须** 遵循 AGENTS.md 中定义的目录结构与项目规范
- **禁止** 引入项目未使用的依赖
- **禁止** 修改核心配置文件（除非明确要求）

### 3. 最小化变更

- **必须** 只修改完成任务所需的最少代码
- **必须** 保持现有功能不受影响
- **禁止** 进行超出需求范围的重构
- **禁止** 删除看似无用但不确定的代码

### 4. 编码前思考

> 灵感来源：[Andrej Karpathy 关于 LLM 编码陷阱的观察](https://x.com/karpathy/status/2015883857489522876)

- **必须** 在实现前明确列出自己的假设，不确定时停下来询问
- **必须** 当需求存在多种合理解读时，呈现各选项及权衡，而非默默选择一种
- **必须** 如果存在更简单的方案，主动提出并推回不合理的复杂度
- **必须** 解释所做更改的原因，告知可能的风险或副作用
- **禁止** 隐藏困惑或重要的实现细节
- **禁止** 在未澄清歧义的情况下直接开始实现

### 5. 简洁优先

- **必须** 用最少的代码解决当前问题，不做推测性开发
- **必须** 检验：资深工程师是否会认为这过于复杂？如果是，简化
- **禁止** 添加需求之外的功能、配置项或"灵活性"
- **禁止** 为仅使用一次的逻辑创建抽象层
- **禁止** 为当前没有证据支持的场景预埋复杂错误处理
- 50 行能解决的问题，不要写成 200 行

### 6. 目标驱动执行

- **必须** 将模糊任务转化为可验证的成功标准再执行
- **必须** 对多步骤任务先制定简要计划，每步附带验证方式
- **建议** 使用测试先行：先写失败测试 → 实现使其通过 → 验证无回归

示例转化：

| 模糊指令 | 可验证目标 |
|---------|-----------|
| "添加表单验证" | "为无效输入编写测试，然后让测试通过" |
| "修复这个 bug" | "编写重现 bug 的测试，然后修复使其通过" |
| "重构 X 模块" | "确保重构前后现有测试都能通过" |

## 安全规则

### 禁止的操作

1. **敏感信息处理**
   - 禁止在代码中硬编码 API 密钥、密码等敏感信息
   - 禁止将敏感信息输出到日志
   - 禁止将敏感数据存储在 localStorage（token 除外）

2. **危险命令**
   - 禁止执行 `rm -rf` 等破坏性命令
   - 禁止修改系统级配置
   - 禁止执行未经用户确认的脚本

3. **网络请求**
   - 禁止向未知第三方发送数据
   - 禁止绕过项目的 API 封装直接请求
   - 禁止在代码中硬编码外部 URL

### 必须遵守的安全实践

1. **输入验证**
   - 所有用户输入必须进行验证
   - API 响应必须进行类型检查
   - URL 参数必须进行编码

2. **错误处理**
   - API 调用必须包含错误处理
   - 错误信息不应暴露敏感细节
   - 异常情况必须有优雅的降级处理

3. **依赖安全**
   - 不引入存在已知漏洞的依赖
   - 建议使用项目已有的工具库
   - 新依赖需要说明必要性

## 交互规范

### 与用户沟通

1. **清晰表达**
   - 使用简洁明了的中文
   - 避免过于技术化的术语
   - 提供具体的代码示例

2. **主动确认**
   - 重大变更前询问用户
   - 不确定需求时请求澄清
   - 提供多个方案时说明各自优缺点

3. **反馈进度**
   - 告知正在执行的操作
   - 完成后总结所做的更改
   - 提示可能需要的后续步骤

### 响应格式

```markdown
## 理解

简述对需求的理解

## 方案

说明将如何实现

## 变更

列出将要修改的文件和内容

## 注意

提示需要注意的事项（如有）
```

## 边界约束

### AI 不应执行的操作

1. **配置变更**
   - 不修改 `vite.config.ts` 核心配置
   - 不修改 `tsconfig.json` 编译选项
   - 不修改 `.env.*` 环境变量文件
   - 不修改 ESLint 规则

2. **架构变更**
   - 不改变项目的目录结构
   - 不替换核心技术栈（如 Vue → React）
   - 不引入新的状态管理方案

3. **发布操作**
   - 不执行 `pnpm publish`
   - 不执行 `git push`
   - 不修改版本号

### 需要用户确认的操作

1. **依赖变更**
   - 添加新的 npm 包
   - 升级现有依赖版本
   - 移除依赖

2. **文件操作**
   - 删除文件
   - 重命名文件
   - 移动文件位置

3. **Git 操作**
   - 创建提交
   - 创建分支
   - 合并代码

---

## 执行优先级

当多个规则冲突时，按以下优先级执行：

1. **安全规则** - 最高优先级，不可违反
2. **核心原则** - 指导所有开发行为
3. **交互规范** - 保证良好的用户体验
4. **边界约束** - 防止意外的破坏性操作
